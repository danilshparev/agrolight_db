# Концепция и проектирование базы данных для ООО «АГРОЛАЙТ»

## Общее описание проекта
Проект посвящен разработке архитектуры реляционной базы данных для автоматизации операционной деятельности компании-трейдера «Агролайт», специализирующейся на оптовой торговле кормовыми добавками для сельскохозяйственных животных. 

Цель разработки — создание информационной системы для минимизации складских издержек, внедрения учета остатков с контролем сроков годности, а также автоматизации документооборота и процессов электронного обмена данными (ЭДО).

## Бизнес-требования и системный анализ
База данных спроектирована на основе формализованных бизнес-требований (БТ) и функциональных требований (ФТ), направленных на устранение рутинных ручных операций. Структура БД приведена к третьей нормальной форме (3NF) и включает следующие ключевые модули:
* **Управление контрагентами и персоналом:** таблицы сотрудников (`Employees`), должностей (`Positions`), клиентов (`Customers`) и контрактов (`Contracts`).
* **Каталог продукции и складской учет:** описание товаров (`Products`), структуры складов (`Warehouses`) и партионного учета на складе (`Storage`) с фиксацией дат производства и сроков годности.
* **Продажи и логистика:** сквозной учет заказов (`Orders`), спецификаций (`Order_items`) и доставок (`Deliveries`).
* **Документооборот (ЭДО):** логирование сопроводительных документов (`Documents`) с контролем статусов подписания.

## Реализованная бизнес-логика и автоматизация

### 1. Представления (Views) для отчетности
Для оптимизации процесса подготовки отчетности по сделкам и остаткам реализованы аналитические витрины:
* `v_stock_balance` — агрегирует данные о текущих остатках товаров на складах, отображает сроки годности и автоматически рассчитывает флаг дефицита номенклатуры.
* `v_sales_by_product` — консолидирует информацию по продажам в разрезе продуктов (общее количество, выручка, число уникальных заказов) для оценки коммерческой эффективности, исключая из расчетов отмененные заказы.

### 2. Триггеры для контроля складских запасов
Реализована автоматическая система предупреждения о дефиците позиций на уровне СУБД. Триггер `trg_check_low_stock` отслеживает изменения в таблице `Storage` при обновлении или добавлении записей. Если текущее количество товара (`quantity`) падает ниже установленного минимального порога (`min_threshold`), система автоматически переводит флаг `low_stock_flag` в значение `TRUE`. Это позволяет оперативно сигнализировать отделу закупок о необходимости пополнения запасов.

### 3. Индексация для оптимизации производительности
Для обеспечения высокой скорости выполнения поисковых запросов и ускорения операций соединения (JOIN) в условиях масштабирования базы данных созданы b-tree индексы по ключевым внешним полям:
* Индексы связей сотрудников и их должностей (`idx_employees_position_id`).
* Индексы складских остатков и складских ячеек (`idx_storage_warehouse_id`, `idx_storage_product_id`).
* Индексы логистического контура и документооборота (`idx_deliveries_order_id`, `idx_documents_order_id`).
